📌 一、核心原则
| 原则 | 说明 | AI协作要求 |
|------|------|------------|
| 契约先行 | 接口变更必须先更新 Swagger 文档 | AI 生成代码前必须确认 Swagger 注释完整 |
| 安全底线 | 任何用户输入必须校验，任何外部回调必须验签 | AI 禁止生成硬编码密钥、明文密码 |
| 可观测性 | 关键操作必留痕，异常必告警 | AI 生成代码必须包含关键日志点 |
| 审核合规 | 隐藏功能逻辑服务端控制 | AI 禁止在前端逻辑中硬编码审核规避逻辑 |
| AI可验证 | 所有代码必须可通过自动化验证 | 每行 AI 生成代码需附带验证注释 |
| 人类终审 | AI 生成代码必须经人工审查 | 核心业务逻辑（资金/邀请）禁止直接合并 |
| 代码一致性 | 强制统一命名、结构体、空值处理 | AI 生成代码必须严格遵循第 5 章规范 |

2.2 统一响应格式
```go
// 响应包装器（所有接口强制使用）
type Response struct {
    Code int         `json:"code"`
    Msg  string      `json:"msg"`
    Data interface{} `json:"data"`
    Ts   int64       `json:"ts"`
}

// 成功示例
func Success(data interface{}) Response {
    return Response{
        Code: 200,
        Msg:  "success",
        Data: data,
        Ts:   time.Now().Unix(),
    }
}

// 错误示例（禁止返回 nil data）
func Error(code int, msg string) Response {
    return Response{
        Code: code,
        Msg:  msg,
        Data: map[string]interface{}{}, // 空对象，非nil
        Ts:   time.Now().Unix(),
    }
}
```

2.3 错误码体系
| 范围 | 含义 | 示例 |
|------|------|------|
| 200 | 成功 | - |
| 4000-4099 | 客户端参数错误 | 4001=邮箱格式错误 |
| 4100-4199 | 业务逻辑错误 | 4101=邀请码不存在 |
| 4200-4299 | 权限/状态错误 | 4201=未完成广告解锁 |
| 5000-5099 | 服务端错误 | 5001=广告回调验签失败 |
| 6000-6099 | 第三方服务错误 | 6001=微信登录超时 |

2.4 Swagger 文档规范
- 所有接口必须通过 swaggo 注释生成 Swagger 文档  
- **必含内容**：接口摘要、详细描述、请求/响应示例、每个参数的类型/必填/说明  
- **参数说明**：Path、Query、Header、Body 需逐项添加 `description` 中文说明  
- **示例**：见本文档「附录 A：Swagger 注释示例」  

✅ 强制要求：  
- 错误信息需用户友好（禁止返回 "SQL error" 等技术细节）  
- 敏感字段脱敏：手机号→138*1234，邮箱→u@qq.com  

📝 三、日志规范
3.1 trace_id 全链路透传（强制）
- 所有业务日志必须包含 `trace_id` 字段，便于链路追踪与问题排查  
- trace_id 由 RequestID 中间件生成，从 context 注入，调用 `pkg/logger.C(ctx)` 自动附带  
- panic 恢复、defer 等场景需从 context 取出 trace_id 显式写入日志，保证异常链路可追溯  
- 日志格式示例：`{"trace_id":"abc123","level":"info","msg":"用户登录成功","user_id":1001,...}`  

3.2 落盘与生命周期管理（config/log.yaml）
```yaml
output: ["file"]
file_path: "/data/logs/{项目名}/app.log"
max_size: 100        # 单文件达100MB触发轮转
max_backups: 30      # 保留最近30个归档文件
max_age: 7           # 超过7天的归档文件自动清理
compress: true       # 轮转后立即gzip压缩
rotation_time: "00:00"
enable_sql_log: false
sensitive_fields: ["password", "id_card", "withdraw_account", "device_id"]
```

3.3 日志级别使用规范
| 级别 | 使用场景 | 示例 |
|------|----------|------|
| DEBUG | 开发调试、细粒度状态 | 参数解析、中间步骤 |
| INFO | 正常业务流程、关键节点 | 请求入口/出口、登录成功、订单创建 |
| WARN | 预期内异常、需关注 | 重试成功、防刷拦截、余额不足 |
| ERROR | 未预期错误、需排查 | 数据库失败、外部调用失败、事务回滚 |

3.4 日志内容规范
- **必含字段**：trace_id、level、msg、timestamp  
- **上下文字段**：user_id、path、duration 等按场景附带  
- **禁止**：打印完整密码、token、银行卡号；单条日志过长（>2KB 建议拆分）  

3.5 关键操作日志清单
| 操作 | 日志级别 | 必含字段 | 脱敏要求 |
|------|----------|----------|----------|
| 用户登录 | info | user_id, login_type, ip, device_fingerprint | device_fingerprint 部分脱敏 |
| 提现申请 | warn | user_id, amount, fee_rate, ip | amount 保留 2 位小数 |
| 事务回滚 | error | operation, user_id, rollback_reason | - |
| 防刷拦截 | warn | action, user_id, ip, reason | - |

3.6 日志输出示例（含 trace_id）
```go
// 通过 context 携带 trace_id，日志库自动注入
logger.C(ctx).Info("用户登录成功",
    zap.Int64("user_id", userID),
    zap.String("login_type", "wechat"),
    zap.String("ip", clientIP),
)
// 输出: {"trace_id":"req-xxx","level":"info","msg":"用户登录成功","user_id":1001,"login_type":"wechat","ip":"1.2.3.4"}
```

🛡️ 四、防刷、防抖与事务规范
4.1 防抖通用工具
```go
// pkg/debounce/debounce.go
func Apply(ctx context.Context, redis *redis.Client, key string, ttl time.Duration) (bool, error) {
    ok, err := redis.SetNX(ctx, key, "1", ttl).Result()
    if err != nil {
        return false, fmt.Errorf("防抖检查失败: %w", err)
    }
    if !ok {
        return false, errors.New("操作进行中，请勿重复提交")
    }
    return true, nil
}
```

4.2 事务标准模板（核心）
```go
func (s *WithdrawService) Process(ctx context.Context, req *WithdrawReq) error {
    tx := s.db.Begin()
    defer func() {
        if r := recover(); r != nil {
            tx.Rollback()
            // 使用 logger.C(ctx) 保证 trace_id 写入，异常链路可追溯
            logger.C(ctx).Error("withdraw panic", zap.Any("recover", r), zap.Stack("stack"))
            panic(r)
        }
    }()
    
    if err := tx.Error; err != nil {
        return fmt.Errorf("事务启动失败: %w", err)
    }
    
    // 1. 审计日志
    if err := tx.Create(&AuditLog{...}).Error; err != nil {
        tx.Rollback()
        return fmt.Errorf("审计日志失败: %w", err)
    }
    
    // 2. 扣减余额（行锁）
    if err := tx.Exec("UPDATE sd_user SET spark = spark - ? WHERE id = ? AND spark >= ?", 
        req.Amount, req.UserID, req.Amount).Error; err != nil {
        tx.Rollback()
        return fmt.Errorf("余额不足: %w", err)
    }
    
    // 3. 创建记录
    if err := tx.Create(&WithdrawRecord{...}).Error; err != nil {
        tx.Rollback()
        return fmt.Errorf("创建记录失败: %w", err)
    }
    
    // 4. 提交前二次校验（税务提示）
    if req.Amount >= 500 {
        tx.Rollback()
        return fmt.Errorf("根据税法规定，单次提现≥500元需自行申报个人所得税")
    }
    
    if err := tx.Commit().Error; err != nil {
        return fmt.Errorf("事务提交失败: %w", err)
    }
    return nil
}
```

📁 五、项目结构与分层架构（框架强制约定）
5.1 标准目录结构（多项目统一）
```
/cmd/
  main.go                 # 入口，仅负责组装与启动
/internal/                # 业务代码，对外不可见
  /handler/               # HTTP 层，参数校验、调用 Service
  /service/               # 业务逻辑层
  /repository/            # 数据访问层，仅与 DB 交互
  /model/                 # PO/DO 定义
  /dto/                   # Request/Response 结构体
/pkg/                     # 可复用公共包（可抽离为独立 go module）
  /logger/                # 日志封装，支持 trace_id 注入
  /middleware/            # 认证、限流、日志等
  /validator/             # 参数校验
  /response/              # 统一响应封装
  /debounce/              # 防抖工具
/config/                  # 配置文件
/docs/                    # Swagger、DB_SCHEMA
/migrations/              # 数据库迁移脚本
```

5.2 分层结构体规范
| 层级 | 命名后缀 | 用途 | 示例 |
|------|----------|------|------|
| DTO | Request/Response | Handler 层输入输出 | WithdrawApplyRequest |
| VO | VO | Service 层跨模块传递 | UserVO |
| PO | 无后缀 | Repository 层 DB 映射 | User（含 password_hash） |
| DO | DO | 领域对象（复杂业务） | WithdrawDO |

✅ 强制要求：  
- 禁止跨层直接传递 PO（DB 模型）  
- VO 必须脱敏（过滤敏感字段）  
- 所有结构体必须有 JSON 标签（含 omitempty）  
- 必填字段必须有 binding 校验标签  

5.3 代码风格与错误处理
```go
// ✅ 正确：错误包装 + 明确上下文
func (s *UserService) GetByID(ctx context.Context, id int64) (UserVO, error) {
    user, err := s.repo.FindByID(ctx, id)
    if err != nil {
        return UserVO{}, fmt.Errorf("查询用户失败 id=%d: %w", id, err)
    }
    return user.ToVO(), nil
}

// ❌ 错误：前端传参决定功能（审核必挂）
// func UnlockFeature(ctx context.Context, feature string) error { ... }
```

5.4 中文注释规范
- **包注释**：每个包首行需有 `// Package xxx 包简介`  
- **公开函数/方法**：必须有中文注释说明用途、参数含义、返回值含义  
- **复杂逻辑**：关键分支、业务规则需行内注释说明  
- **结构体字段**：对外 API 的 Request/Response 字段建议添加 `example` 标签供 Swagger 展示，并辅以行内注释  

```go
// WithdrawApplyRequest 提现申请请求体
type WithdrawApplyRequest struct {
    Amount float64 `json:"amount" binding:"required,gt=0" example:"100.00"` // 提现金额，单位元
    // ...
}
```

🔄 九、启动与优雅关闭
- 启动：按依赖顺序初始化（Config → Logger → DB → Redis → Router）  
- 关闭：监听 SIGTERM/SIGINT，停止接收新请求，等待进行中请求完成（超时 30s 强制退出）  
- 资源释放：依次关闭 HTTP Server、DB 连接池、Redis、MQ
