方案模式：预生成订单号 + 前端轮询确认
核心目标：确保奖励发放安全（防刷），优化用户等待体验（解决回调延迟）。

二、接口定义规范 (后端研发主导，前端配合)
后端需提供以下 3 个核心接口 给前端调用：
1. 创建预发奖订单接口 (Pre-Order)
调用时机：用户点击“观看广告获取奖励”按钮，广告加载前。
输入参数：user_id, ad_position_code (广告位类型), reward_type (奖励类型), reward_amount (数量)。
输出参数：
order_no (String): 全局唯一订单号 (核心字段，用于关联后续流程)。
status: 初始状态为 PENDING (待确认)。
expire_time: 订单有效期（如 5 分钟）。
后端逻辑：
生成唯一 order_no (建议使用 UUID 或 雪花算法)。
在数据库创建一条记录：{order_no, user_id, amount, status=PENDING, create_time}。
将 order_no 返回给前端。
2. 查询订单状态接口 (Query Status)
调用时机：广告播放结束后，前端开始轮询调用。
输入参数：order_no。
输出参数：
status: 枚举值 (PENDING: 等待回调, SUCCESS: 已发奖, FAILED: 失败/过期)。
message: 提示信息 (如 "奖励已到账", "正在确认中")。
current_balance: (可选) 用户当前最新余额。
后端逻辑：
根据 order_no 查库，返回当前状态。
注意：此接口只读，不修改数据。
3. (内部接口) 穿山甲回调接收接口 (Callback Receiver)
调用方：穿山甲服务器 (非前端调用)。
输入参数：trans_id, user_id (透传), sign, timestamp 等。
关键逻辑：
验签：使用 Token 验证签名。
关联订单：这里有个难点，穿山甲回调通常带的是 user_id 而不是我们生成的 order_no。
解决方案 A (推荐)：前端在请求广告时，将 order_no 作为 custom_data 或 user_id 的一部分透传给穿山甲（如果穿山甲支持透传自定义字符串）。强烈建议将 order_no 直接作为 user_id 透传，或者拼接在 user_id 后（如 uid_123_order_xyz），回调时解析出来。
解决方案 B (备选)：如果只能透传固定 user_id，后端需查询该用户最近一个 PENDING 状态的订单进行匹配（需处理并发和多订单情况，较复杂）。
更新状态：验签通过后，将数据库中对应 order_no 的状态更新为 SUCCESS，并执行真正的加币操作。
幂等性：检查 trans_id 是否已处理，防止重复发奖。
三、前后端交互流程详解 (研发执行步骤)
步骤 1: 初始化与预下单
前端：用户点击“免费领金币”。
前端 -> 后端：调用 [接口 1: 创建预发奖订单]。
后端：生成 order_no = "ORD_20260228_001"，存入 DB (状态 PENDING)，返回给前端。
前端：拿到 order_no。
前端 (SDK)：调用穿山甲 SDK 请求广告。
关键动作：在 SDK 配置中，将 order_no 填入 user_id 字段（或 custom_data 字段，视穿山甲具体版本支持情况而定，务必确认穿山甲回调能原样返回这个值）。
注：如果穿山甲强制要求 user_id 必须是数字且不能变，则需在 custom_data 中透传 order_no，并在回调解析 custom_data。
步骤 2: 广告播放
穿山甲：返回广告素材。
前端：播放视频，用户观看。
前端：监听 onAdShow (开始), onAdComplete (播放完成), onAdClose (关闭)。
步骤 3: 轮询等待 (核心体验环节)
前端：当监听到 onAdComplete 或 onAdClose 后：
显示 Loading 弹窗：“正在核对奖励...”。
启动定时器，每隔 1.5 秒 调用一次 [接口 2: 查询订单状态]，传入 order_no。
后端：接收查询，返回 DB 中的状态（此时若穿山甲回调未到，返回 PENDING）。
前端：
若返回 PENDING：继续轮询。
若返回 SUCCESS：停止轮询，关闭 Loading，提示“奖励已到账”，刷新 UI 余额。
若返回 FAILED：停止轮询，提示“领取失败，请稍后重试”。
超时保护：若轮询超过 30 秒 (或 20 次) 仍为 PENDING，停止轮询。提示用户“系统繁忙，奖励将在稍后自动到账，请留意钱包变动”，并允许用户离开页面（后端收到回调后依然会入账，用户下次打开 App 能看到）。
步骤 4: 异步回调入账
穿山甲：判定有效观看，向 后端 Callback URL 发送 POST 请求。
参数中包含之前透传的 order_no (在 user_id 或 custom_data 中)。
后端：
解析出 order_no。
验签成功。
查找 DB 中该 order_no 的记录。
更新状态为 SUCCESS，执行 balance = balance + amount。
记录日志。
返回 {"ret": "success"} 给穿山甲。
四、异常场景与处理策略 (QA 测试重点)
表格
异常场景	现象	处理策略	责任方
回调延迟	广告看完了，过了 10 秒回调才到。	前端轮询机制覆盖此场景。只要轮询时间 > 延迟时间，用户无感知。	前后端配合
回调丢失	穿山甲因网络问题未发送回调（极少见，穿山甲有重试机制）。	1. 后端记录所有 PENDING 超过 24 小时的订单。
2. 运营人工核对或开发定时任务二次查询穿山甲 API 补单。	后端
用户中途退出	轮询到一半用户杀进程或断网。	后端回调依然会执行，状态会变更为 SUCCESS。用户下次进入 App 查询余额时可见。需在前端“消息中心”推送一条“奖励已到账”通知。	前端 + 后端
重复点击	用户快速连续点击“看广告”。	前端做防抖（Debounce）；后端检查该用户是否有未完成的 PENDING 订单，若有则拒绝创建新订单，提示“上一个奖励正在确认中”。	前端 + 后端
签名伪造	黑客模拟穿山甲请求回调接口。	后端严格校验 sign 签名，失败直接丢弃并报警。	后端
透传失败	穿山甲回调中没拿到 order_no。	高风险。需在测试环境充分验证透传字段（user_id 或 custom_data）的完整性。若丢失，需通过 user_id + 时间窗口 模糊匹配最近一笔订单（需后端增加此容错逻辑）。	前端 + 后端

💡 给研发的特别提示 (Tips)
透传字段选择：请务必先查阅最新版穿山甲文档，确认 user_id 字段是否支持字符串（因为 order_no 通常含字母）。如果 user_id 仅限数字，必须使用 custom_data 字段透传 order_no，并在回调解析 custom_data。这是对接成败的关键。
日志留存：后端必须记录穿山甲回调的原始报文至少 30 天，以便出现客诉时排查是穿山甲没发、发了被拦截、还是解析错误。
测试环境：不要直接用线上账号测试，务必在穿山甲后台将测试手机加入白名单，使用测试广告（带有“广告”水印），以免产生真实计费或被风控。
